# BIP Deck Platform — Data Model (Phase 1)

**Version:** 0.1
**Status:** Living document, companion to architecture and phasing docs.
**Authoritative on:** Phase 1 Postgres schema and Prisma models.

---

## 0. About this document

Companion to `bip-deck-platform-architecture.md` and `bip-deck-platform-phasing.md`. Defines the concrete Postgres schema for Phase 1 as Prisma models. Hand this to Copilot when scaffolding the database layer.

Phase 2+ entities (brand kits, view events, viewer sessions, mention join tables) are sketched in §5 but not detailed here — they'll get their own additions when those phases approach. Phase 1 ships with the eleven tables in §3, and the schema is designed so Phase 2/3 extensions are additive, not migrations of existing data.

---

## 1. Conventions

- **Database:** Postgres 16
- **ORM:** Prisma 5+
- **Primary keys:** UUIDs (v4 generated by Prisma's `@default(uuid())`). UUIDv7 considered for future if write volume justifies sortable keys.
- **Naming:** `snake_case` in the database, `camelCase` in the Prisma client. Every model maps with `@@map` and every multi-word field with `@map`.
- **Timestamps:** Every table has `created_at`. Mutable tables also have `updated_at` (Prisma-managed via `@updatedAt`).
- **Soft delete:** `User` and `Deck` only. Everything else hard-deletes via foreign-key cascade when its parent goes.
- **Enums:** Postgres native enums for stable enumerations. String columns for things expected to grow.
- **Indexes:** Every foreign key gets an index. Compound indexes are added per documented query pattern, not speculatively.

---

## 2. Entity overview

Phase 1 has eleven tables in three groups:

- **Identity** — `User`, `Session`, `ShareLinkRecipient`
- **Content** — `Deck`, `DeckVersion`, `Comment`, `Vote`
- **Workflow** — `ShareLink`, `Job`, `AIConversation`, `AIMessage`

Relationships at a glance:

```
User ──< Session
User ──< Deck (createdBy)
User ──< Comment (authorUser)
User ──< Vote (user)
User ──< ShareLink (createdBy)
User ──< Job (createdBy)
User ──< AIConversation (createdBy)
User ──< DeckVersion (createdBy)

Deck ──< DeckVersion
Deck ──< Comment
Deck ──< ShareLink
Deck ──< AIConversation
Deck ──< Job (nullable, set null on delete)

ShareLink ──< ShareLinkRecipient

ShareLinkRecipient ──< Comment (authorRecipient)
ShareLinkRecipient ──< Vote (recipient)

Comment ──< Vote
Comment ──< Comment (parent/replies, self-relation)

AIConversation ──< AIMessage
AIMessage ──> Job (related, nullable)
Job ──< AIMessage
```

## 3. Entities

### 3.1 User

Anyone who can authenticate into the admin portal. Phase 1: BIP team only. Phase 3+ extends via the `kind` field for client stakeholders.

```prisma
model User {
  id           String    @id @default(uuid()) @db.Uuid
  email        String    @unique
  name         String
  passwordHash String?   @map("password_hash")
  kind         UserKind  @default(TEAM)
  createdAt    DateTime  @default(now()) @map("created_at")
  updatedAt    DateTime  @updatedAt @map("updated_at")
  lastLoginAt  DateTime? @map("last_login_at")
  deletedAt    DateTime? @map("deleted_at")

  sessions            Session[]
  decksCreated        Deck[]            @relation("DeckCreator")
  comments            Comment[]
  votes               Vote[]
  shareLinksCreated   ShareLink[]
  deckVersionsCreated DeckVersion[]
  aiConversations     AIConversation[]
  jobsCreated         Job[]

  @@map("users")
}

enum UserKind {
  TEAM
  // CLIENT added in Phase 3
}
```

**Notes**

- `passwordHash` is nullable for forward compatibility. Phase 4 SSO users will have null hash.
- `kind` defaults to TEAM and Phase 3 adds CLIENT — no migration impact on existing rows.
- Soft delete supported but rarely used for team users; deactivation (setting a flag) is the typical flow.

### 3.2 Session

Authentication session. 90-day rolling expiry. The token plaintext lives in an HTTP-only cookie on the client; the database stores a hash.

```prisma
model Session {
  id         String   @id @default(uuid()) @db.Uuid
  userId     String   @map("user_id") @db.Uuid
  tokenHash  String   @unique @map("token_hash")
  expiresAt  DateTime @map("expires_at")
  lastUsedAt DateTime @default(now()) @map("last_used_at")
  userAgent  String?  @map("user_agent")
  ipAddress  String?  @map("ip_address")
  createdAt  DateTime @default(now()) @map("created_at")

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId])
  @@index([expiresAt])
  @@map("sessions")
}
```

**Notes**

- Token hashed server-side with HMAC-SHA256 using a secret from environment.
- A cron job prunes expired sessions weekly.
- `ipAddress` and `userAgent` are recorded for audit only; never used for binding (would break legitimate roaming).

### 3.3 Deck

The top-level entity. One deck = one presentation = one git repository = one filesystem directory.

```prisma
model Deck {
  id             String         @id @default(uuid()) @db.Uuid
  slug           String         @unique
  title          String
  lifecycleStage LifecycleStage @default(DRAFT) @map("lifecycle_stage")
  repoPath       String         @unique @map("repo_path")
  headCommitSha  String?        @map("head_commit_sha")
  createdById    String         @map("created_by_id") @db.Uuid
  createdAt      DateTime       @default(now()) @map("created_at")
  updatedAt      DateTime       @updatedAt @map("updated_at")
  archivedAt          DateTime?      @map("archived_at")
  deletedAt           DateTime?      @map("deleted_at")
  editingUserId       String?        @map("editing_user_id") @db.Uuid
  editingHeartbeatAt  DateTime?      @map("editing_heartbeat_at")

  createdBy       User             @relation("DeckCreator", fields: [createdById], references: [id])
  versions        DeckVersion[]
  comments        Comment[]
  shareLinks      ShareLink[]
  aiConversations AIConversation[]
  jobs            Job[]

  @@index([createdById])
  @@index([deletedAt])
  @@map("decks")
}

enum LifecycleStage {
  OUTLINE
  DRAFT
  REVIEWING
  FINAL
}
```

**Notes**

- `slug` is the human-readable URL segment (`acme-series-a`). Unique across all decks including soft-deleted ones to prevent slug recycling collisions.
- `repoPath` is the filesystem path of the deck's git repo. Application enforces uniqueness; the DB constraint is a safety net.
- `headCommitSha` is a denormalized pointer to the main-branch tip. Updated on each commit. The git repo on disk is the authoritative source; this column is a cache for fast reads.
- Soft delete via `deletedAt`. A cron hard-deletes records older than 30 days, cascading to dependent records and removing the git repo from disk.
- `archivedAt` is a softer state: user-initiated, preserved indefinitely, read-only, hidden from default deck list.
- `editingUserId` and `editingHeartbeatAt` implement a soft edit lock for the AI editor. A lock is held when the heartbeat is fresh (within 2 minutes). See `bip-deck-platform-ai-editor.md` §9 for the full behavior.

### 3.4 DeckVersion

A tagged snapshot. Phase 1: auto-created on transition to FINAL; can also be manually created.

```prisma
model DeckVersion {
  id          String   @id @default(uuid()) @db.Uuid
  deckId      String   @map("deck_id") @db.Uuid
  commitSha   String   @map("commit_sha")
  tagName     String   @map("tag_name")
  notes       String?
  createdById String   @map("created_by_id") @db.Uuid
  createdAt   DateTime @default(now()) @map("created_at")

  deck      Deck @relation(fields: [deckId], references: [id], onDelete: Cascade)
  createdBy User @relation(fields: [createdById], references: [id])

  @@unique([deckId, tagName])
  @@index([deckId])
  @@map("deck_versions")
}
```

**Notes**

- Each row points to a git tag of the same name in the deck's repo.
- `tagName` unique within a deck (`final-v1`, `final-v2`). Auto-generated on FINAL transition; editable.
- Snapshot share links (Phase 3) reference a `commit_sha` directly, not a `DeckVersion` row, so versions can be deleted without breaking links.

### 3.5 Comment

Feedback on a slide. Phase 1: slide-level only. Phase 2: element-level via the `elementAnchor` JSON column already in the schema.

```prisma
model Comment {
  id                 String         @id @default(uuid()) @db.Uuid
  deckId             String         @map("deck_id") @db.Uuid
  slideId            String         @map("slide_id")
  elementAnchor      Json?          @map("element_anchor")
  authorUserId       String?        @map("author_user_id") @db.Uuid
  authorRecipientId  String?        @map("author_recipient_id") @db.Uuid
  authorDisplayName  String         @map("author_display_name")
  body               String
  parentId           String?        @map("parent_id") @db.Uuid
  status             CommentStatus  @default(OPEN)
  adminNote          String?        @map("admin_note")
  createdAt          DateTime       @default(now()) @map("created_at")
  updatedAt          DateTime       @updatedAt @map("updated_at")

  deck            Deck                @relation(fields: [deckId], references: [id], onDelete: Cascade)
  authorUser      User?               @relation(fields: [authorUserId], references: [id])
  authorRecipient ShareLinkRecipient? @relation(fields: [authorRecipientId], references: [id])
  parent          Comment?            @relation("CommentReplies", fields: [parentId], references: [id], onDelete: Cascade)
  replies         Comment[]           @relation("CommentReplies")
  votes           Vote[]

  @@index([deckId, slideId])
  @@index([deckId, status])
  @@index([parentId])
  @@map("comments")
}

enum CommentStatus {
  OPEN
  IN_REVIEW
  PLANNED
  DONE
  DISMISSED
}
```

**Notes**

- `authorUserId` XOR `authorRecipientId` — exactly one must be set. Enforced in application; a Postgres CHECK constraint adds a safety net (see §4.3).
- `authorDisplayName` is denormalized for display. Mirrors `User.name` for team authors; mirrors the name a recipient entered on first visit otherwise. Avoids joining for every comment render.
- `elementAnchor` is null in Phase 1. Phase 2 populates with something like `{ selector, x, y, rect }` for pin-on-element comments.
- Parent must be on the same slide (existing tool's rule). Application-enforced; not a DB constraint because it would require a trigger.

### 3.6 Vote

One row per (comment, voter). Direction is +1 or -1.

```prisma
model Vote {
  id          String   @id @default(uuid()) @db.Uuid
  commentId   String   @map("comment_id") @db.Uuid
  userId      String?  @map("user_id") @db.Uuid
  recipientId String?  @map("recipient_id") @db.Uuid
  direction   Int
  createdAt   DateTime @default(now()) @map("created_at")

  comment   Comment             @relation(fields: [commentId], references: [id], onDelete: Cascade)
  user      User?               @relation(fields: [userId], references: [id])
  recipient ShareLinkRecipient? @relation(fields: [recipientId], references: [id])

  @@unique([commentId, userId])
  @@unique([commentId, recipientId])
  @@index([commentId])
  @@map("votes")
}
```

**Notes**

- Same XOR pattern as `Comment`: one of `userId` or `recipientId` set.
- Two partial unique indexes enforce "one vote per voter per comment." Postgres treats NULL as non-restrictive in unique constraints, so this works without conflicting with the XOR.
- Vote tally is computed on read: `SUM(direction) WHERE comment_id = ?`. Cheap with the index.
- `direction` constrained to `-1` or `1` via CHECK constraint (§4.3).

### 3.7 ShareLink

Issued access to a deck. Phase 1: only one audience type (REVIEWER); Phase 3 adds the others.

```prisma
model ShareLink {
  id                String         @id @default(uuid()) @db.Uuid
  deckId            String         @map("deck_id") @db.Uuid
  token             String         @unique
  audienceType      AudienceType   @map("audience_type")
  versionBinding    VersionBinding @default(LIVE) @map("version_binding")
  boundCommitSha    String?        @map("bound_commit_sha")
  createdById       String         @map("created_by_id") @db.Uuid
  createdAt         DateTime       @default(now()) @map("created_at")
  expiresAt         DateTime?      @map("expires_at")
  revokedAt         DateTime?      @map("revoked_at")
  lastUsedAt        DateTime?      @map("last_used_at")
  downloadsDisabled Boolean        @default(false) @map("downloads_disabled")

  deck       Deck                 @relation(fields: [deckId], references: [id], onDelete: Cascade)
  createdBy  User                 @relation(fields: [createdById], references: [id])
  recipients ShareLinkRecipient[]

  @@index([deckId])
  @@map("share_links")
}

enum AudienceType {
  REVIEWER
  // COLLABORATOR added in Phase 3 (client portal access)
  // PUBLIC_VIEWER, EMAIL_GATED_VIEWER, CODED_VIEWER added in Phase 3
}

enum VersionBinding {
  LIVE
  SNAPSHOT
}
```

**Notes**

- `token` is an opaque random string (32 bytes, base64url encoded), generated server-side. Lookup by token is the hot path; the unique index covers it.
- `versionBinding` defaults to LIVE. SNAPSHOT requires `boundCommitSha` — enforced via CHECK constraint (§4.3).
- Phase 1 only emits REVIEWER links; the enum is structured so Phase 3 additions are non-breaking.
- Active link predicate: `revoked_at IS NULL AND (expires_at IS NULL OR expires_at > NOW())`.

### 3.8 ShareLinkRecipient

The lightweight identity of someone who has visited a share link. Created on first visit when the recipient claims the link with a name.

```prisma
model ShareLinkRecipient {
  id             String   @id @default(uuid()) @db.Uuid
  shareLinkId    String   @map("share_link_id") @db.Uuid
  email          String?
  displayName    String   @map("display_name")
  clientId       String   @unique @map("client_id") @db.Uuid
  firstVisitedAt DateTime @default(now()) @map("first_visited_at")
  lastVisitedAt  DateTime @default(now()) @map("last_visited_at")
  visitCount     Int      @default(1) @map("visit_count")

  shareLink ShareLink @relation(fields: [shareLinkId], references: [id], onDelete: Cascade)
  comments  Comment[]
  votes     Vote[]

  @@index([shareLinkId])
  @@map("share_link_recipients")
}
```

**Notes**

- `clientId` is a UUID generated client-side, persisted in `localStorage` plus a cookie. On revisit the client sends it back so we resolve to the same recipient. This matches the existing tool's behavior.
- `email` is nullable in Phase 1 (reviewer links require name only). Phase 3 email-gated viewer links populate it.
- Email is not unique — the same person may receive different share links across multiple decks. Identity is per-link, not per-person.

### 3.9 Job

Async work record. Phase 1 has no queue; jobs are created and run synchronously, but they're persisted so the AI editor has a history and Phase 3 can swap in BullMQ without schema change.

```prisma
model Job {
  id            String    @id @default(uuid()) @db.Uuid
  deckId        String?   @map("deck_id") @db.Uuid
  kind          JobKind
  status        JobStatus @default(QUEUED)
  createdById   String    @map("created_by_id") @db.Uuid
  label         String?
  input         Json
  output        Json?
  error         String?
  workingBranch String?   @map("working_branch")
  startedAt     DateTime? @map("started_at")
  completedAt   DateTime? @map("completed_at")
  createdAt     DateTime  @default(now()) @map("created_at")
  updatedAt     DateTime  @updatedAt @map("updated_at")

  deck      Deck?       @relation(fields: [deckId], references: [id], onDelete: SetNull)
  createdBy User        @relation(fields: [createdById], references: [id])
  messages  AIMessage[]

  @@index([deckId])
  @@index([status])
  @@map("jobs")
}

enum JobKind {
  AI_EDIT
  // OUTLINE_GENERATION, BRAND_EXTRACTION added in Phase 2
  // TRIAGE_SLIDE, TRIAGE_ROLLUP, AGENTIC_EDIT, PDF_EXPORT added in Phase 3
}

enum JobStatus {
  QUEUED
  RUNNING
  AWAITING_REVIEW
  DONE
  FAILED
  CANCELED
}
```

**Notes**

- `deckId` is nullable with SET NULL on cascade — job records survive deck hard-deletes for audit.
- `workingBranch` is set on AI_EDIT jobs to track which git branch holds the proposed diff. After accept/reject, the branch is merged or deleted; the job record stays.
- `input` and `output` are JSON; structure is per-kind. Application defines TypeScript types per kind.
- Phase 1 transitions: QUEUED → RUNNING → AWAITING_REVIEW → DONE (on accept) / CANCELED (on reject) / FAILED.

### 3.10 AIConversation and AIMessage

Per-deck chat history with the AI editor. One conversation belongs to one deck; messages chain within a conversation.

```prisma
model AIConversation {
  id          String             @id @default(uuid()) @db.Uuid
  deckId      String             @map("deck_id") @db.Uuid
  createdById String             @map("created_by_id") @db.Uuid
  kind        AIConversationKind @default(EDITOR)
  title       String?
  approvedAt  DateTime?          @map("approved_at")
  createdAt   DateTime           @default(now()) @map("created_at")
  updatedAt   DateTime           @updatedAt @map("updated_at")

  deck      Deck        @relation(fields: [deckId], references: [id], onDelete: Cascade)
  createdBy User        @relation(fields: [createdById], references: [id])
  messages  AIMessage[]

  @@index([deckId])
  @@index([deckId, kind])
  @@map("ai_conversations")
}

model AIMessage {
  id             String      @id @default(uuid()) @db.Uuid
  conversationId String      @map("conversation_id") @db.Uuid
  role           MessageRole
  content        Json
  model          String?
  tokensIn       Int?        @map("tokens_in")
  tokensOut      Int?        @map("tokens_out")
  costCents      Int?        @map("cost_cents")
  relatedJobId   String?     @map("related_job_id") @db.Uuid
  createdAt      DateTime    @default(now()) @map("created_at")

  conversation AIConversation @relation(fields: [conversationId], references: [id], onDelete: Cascade)
  relatedJob   Job?           @relation(fields: [relatedJobId], references: [id], onDelete: SetNull)

  @@index([conversationId])
  @@map("ai_messages")
}

enum MessageRole {
  USER
  ASSISTANT
  SYSTEM
}

enum AIConversationKind {
  EDITOR
  OUTLINE
}
```

**Notes**

- `content` is JSON to accommodate structured tool calls and multi-part messages in Phase 3 (agentic mode). Phase 1 stores `{ text }` for editor turns; outline turns use richer shapes (see `apps/web/lib/outline/service.ts` — `{ kind: 'brief'|'user'|'assistant'|'assistant_error', ... }`).
- `title` is null on creation; auto-generated by a cheap model on first user message (e.g. "Tightening slide 7 hero"). For outline conversations it is seeded with the deck title.
- `relatedJobId` ties a message to the job it spawned. Lets the UI render "this message produced this edit" affordances.
- `kind` distinguishes the AI editor chat (`EDITOR`, default) from the Phase 2.5 outline-first authoring flow (`OUTLINE`). Editor queries filter `kind = EDITOR` so outline conversations don't leak into the editor UI. There is at most one OUTLINE conversation per deck.
- `approvedAt` is set only on OUTLINE conversations when the user clicks "Approve" — that scaffolds slide stubs into the deck repo and locks the outline against further edits. Always null for EDITOR.

## 4. Cross-cutting

### 4.1 Soft delete

Only `User` and `Deck` support soft delete. Every application query that returns these tables filters `WHERE deleted_at IS NULL` by default; a Prisma middleware enforces this so it can't be forgotten. A weekly cron hard-deletes records older than 30 days, cascading to dependents and (for decks) removing the git repo on disk.

### 4.2 Cascade rules summary

| Action                           | Effect                                                                                                       |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| Deck hard-deleted                | CASCADE to comments, votes, share links, share link recipients, deck versions, AI conversations, AI messages |
| Deck hard-deleted                | SET NULL on jobs (preserved for audit)                                                                       |
| Comment hard-deleted             | CASCADE to votes, reply comments                                                                             |
| Session expires or user logs out | DELETE the session row                                                                                       |
| User hard-deleted                | CASCADE to sessions; SET NULL on authored content (preserves history with anonymized author label)           |

Phase 1 has no UI for hard-deleting users; the cascade rules are designed for completeness.

### 4.3 Constraints not expressible in Prisma

These need a raw SQL migration applied after the Prisma migration:

```sql
-- Comment author XOR
ALTER TABLE comments
  ADD CONSTRAINT comment_exactly_one_author
  CHECK ((author_user_id IS NOT NULL)::int + (author_recipient_id IS NOT NULL)::int = 1);

-- Vote voter XOR
ALTER TABLE votes
  ADD CONSTRAINT vote_exactly_one_voter
  CHECK ((user_id IS NOT NULL)::int + (recipient_id IS NOT NULL)::int = 1);

-- Vote direction valid
ALTER TABLE votes
  ADD CONSTRAINT vote_direction_valid
  CHECK (direction IN (-1, 1));

-- Snapshot share link has a commit
ALTER TABLE share_links
  ADD CONSTRAINT share_link_snapshot_has_commit
  CHECK (version_binding = 'LIVE' OR bound_commit_sha IS NOT NULL);
```

The "comment parent must be on the same slide" rule is enforced in application code only; a trigger would be over-engineering for Phase 1.

### 4.4 Indexes summary

| Table                   | Indexes                                                                                                          |
| ----------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `users`                 | unique on `email`                                                                                                |
| `sessions`              | unique on `token_hash`, index on `user_id`, index on `expires_at`                                                |
| `decks`                 | unique on `slug`, unique on `repo_path`, index on `created_by_id`, index on `deleted_at`                         |
| `deck_versions`         | unique on `(deck_id, tag_name)`, index on `deck_id`                                                              |
| `comments`              | index on `(deck_id, slide_id)`, index on `(deck_id, status)`, index on `parent_id`                               |
| `votes`                 | unique partial on `(comment_id, user_id)`, unique partial on `(comment_id, recipient_id)`, index on `comment_id` |
| `share_links`           | unique on `token`, index on `deck_id`                                                                            |
| `share_link_recipients` | unique on `client_id`, index on `share_link_id`                                                                  |
| `jobs`                  | index on `deck_id`, index on `status`                                                                            |
| `ai_conversations`      | index on `deck_id`; compound `(deck_id, kind)` for filtering editor vs outline lookups                              |
| `ai_messages`           | index on `conversation_id`                                                                                       |

### 4.5 Migrations

Prisma Migrate handles schema changes. Phase 1 ships one initial migration creating all ten tables, plus one post-Prisma raw SQL file applying the CHECK constraints in §4.3. Each subsequent phase's data model update will be additive — new tables, new columns with safe defaults, no destructive changes.

## 5. Phase 2+ entities (preview)

Designed for in Phase 1 schema decisions but not implemented yet:

- **`BrandKit`, `BrandKitVersion`, `BrandToken`, `BrandPattern`, `BrandReference`** — Phase 2. `Deck` gains a nullable `brand_kit_version_id`.

When those phases approach, this doc gains §6, §7, etc. — it's extended, not rewritten.
